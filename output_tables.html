Output table fields & valuesÔÉÅ

To serve OCTOPUS database collections as geospatial layers via an interactive map Web interface and to allow for data manipulation via the Web Feature Service protocol, data sub-collections are served to GeoServer as a flat data tables, which are described in detail below 1.
When downloading data from OCTOPUS, users are presented with point or polygon geospatial data files with associated attribute tables. Direct connections to the PostgreSQL/PostGIS database are possible upon request. Munack and Codilean (2022) provide a complete documentation of the relational database, including a detailed database model diagram and searchable HTML documentation generated using SchemaSpy (https://schemaspy.org, last access: 21 Aug 2023).

The above section is a modified version of Section 3 from Codilean et al. 2022

CRN outputÔÉÅ

OCTOPUS CRN Be-10 and Al-26 data have been recalculated 2 so that nuclide concentrations and denudation rates are internally consistent and comparable. These and related data, including concentrations and denudation rates as published, are served as geospatial vector layers (see CRN vector spatial data).
In addition, OCTOPUS CRN collections also provide CRN ancillary data comprising raster layers, including a digital elevation model, gradient raster, flow direction and flow accumulation rasters, atmospheric pressure raster, and CRN production scaling and topographic shielding factor rasters, as well as a set of structured input data files (see ‚ÄòCSV‚Äô folder subsection) for the purpose of data recalculation integrity. The idea behind the above is that, since the CAIRN algorithm is automated (see LSDTopoTools documentation), users can simply publish a DEM of their study area, CRN data files, and CAIRN input files, and denudation rates should be reproducible.

CRN ancillary dataÔÉÅ

‚ÄòCSV‚Äô folderÔÉÅ

  • s‚Ä¶_CRNData.csv ‚Äì CAIRN CRN concentrations input file (see LSDTopoTools documentation)

  • s‚Ä¶_CRNRasters.csv ‚Äì Path to raster files (see LSDTopoTools documentation)

  • s‚Ä¶_CRNResults.csv ‚Äì Original CAIRN output (= results)

‚ÄòRaster‚Äô folderÔÉÅ

The raster data uses the WGS84/UTM projected coordinate reference system, UTM zones depending on the extent and location of each data package.

  • s‚Ä¶_atmospres.bil ‚Äì An atmospheric pressure raster, showing local atmospheric pressure in hPa calculated based on the NCEP2 climate reanalysis data (Compo et al., 2011)

  • s‚Ä¶_atmospres.hdr ‚Äì s‚Ä¶_atmospres.bil header file

  • s‚Ä¶_d8flowdir.bil ‚Äì A flow-direction raster calculated using the D8 flow-routing method (Jenson and Domingue, 1988)

  • s‚Ä¶_d8flowdir.hdr ‚Äì s‚Ä¶_d8flowdir.bil header file

  • s‚Ä¶_demhydro.bil ‚Äì A hydrologically corrected DEM with elevation values in metres (raster)

  • s‚Ä¶_demhydro.hdr ‚Äì s‚Ä¶_demhydro.bil header file

  • s‚Ä¶_flowacc.bil ‚Äì A flow-accumulation raster calculated with the same D8 method as used for d8flowdir

  • s‚Ä¶_flowacc.hdr ‚Äì s‚Ä¶_flowacc.bil header file

  • s‚Ä¶_gradmkm.bil ‚Äì A slope gradient raster calculated using the method described in Horn (1981) with units in m km-1

  • s‚Ä¶_gradmkm.hdr ‚Äì s‚Ä¶_gradmkm.bil header file

  • s‚Ä¶_prodscale.bil ‚Äì A cosmogenic nuclide production scaling raster calculated using the method described in Stone (Stone, 2000)

  • s‚Ä¶_prodscale.hdr ‚Äì s‚Ä¶_prodscale.bil header file

  • s‚Ä¶_toposhield.bil ‚Äì A cosmogenic nuclide production topographic shielding raster calculated using the method described in Codilean (2006)

  • s‚Ä¶_toposhield.hdr ‚Äì s‚Ä¶_toposhield.bil header file

Note

Raster files are provided in .bil (Band Interleaved by Line) image encoding format, which cannot be considered an image format itself, but is depended on the accompanying header (.hdr) file meaning that users will always need the pair of files with one and the same filename.
